REST

  • An architectural style for distributed applications, consisting of several constraints
  • Unless all constraints are followed, REST purists will argue that its not a REST api but HTTP over JSON
  • Protocol and platform independant, but usually REST over HTTP. Builds upon HTTP insted of reinventing the wheel
  • REST, RESTful web services, REST web services

REST constraints

  • Client-server - separation of concerns
  • Stateless, as in all session state stored on the client
  • Responses should be cacheable where possible to improve performance
  • Layered to reduce complexity and promote independence of each component
  • Uniform interface for simplicity

Interface constraints

  • Identification of resources - "the nouns"
  • Manipulation of resources - "the verbs"
  • Self-descriptive messages and resource representations
  • Hypermedia-driven

Resources

  • The nouns: accounts, cities, people, documents
  • Are manipulated by verbs
  • Has its own address: http://server.com/api/accounts
  • May be found in multiple representations, e.g.
    • http://server.com/api/accounts.json
    • http://server.com/api/accounts.pdf

Tips

Final recommendations

  • Purist vs. pragmatist (choose the latter)
  • A native app client has different attack surface than a web application. Think of security in server context and client context.
  • Research and apply security measures before going to production
  • Think forward. What will happen to existing clients if you update the API?

Final recommendations

  • Keep resources light. Move business logic to business layer
  • Allow yourself to spend some time designing the API
  • Design as if you were to support three different clients or more. This improves API usability and separation of concerns
